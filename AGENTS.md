# UNREALの記事追加ワークフロー

ユーザーが「新しい記事を追加したい」「記事を追加して」「本番記事を登録したい」など、
UNREALへの記事登録を依頼した場合は、**すぐにコードや記事データを編集してはならない**。
まず、このワークフローに従って必要な情報を対話的に収集する。

## 対話の原則

- 一度にすべての情報を質問しない。回答を受けてから、次の1項目だけを質問する。
- 最初の質問は必ず次の文言にする。

  `新しい記事を登録します。まず記事タイトルを教えてください。`

- 必須情報を受け取ったら、次の順で1項目ずつ確認する。

  1. 記事タイトル
  2. 記事概要（次の質問文：`記事概要を3〜4行で教えてください。`）
  3. エリア
  4. タグ
  5. 通常価格
  6. 販売価格
  7. Codocの記事URL
  8. サムネイル画像

- エリアの質問では、その時点で `src/content/articles.ts` の `areas` に登録されている選択肢を提示する。現在は `全国`、`東京`、`大阪`、`名古屋`、`福岡`、`札幌`、`仙台`、`神奈川`、`千葉`。ただし記事の `area` には `全国` を登録しない。
- タグの質問では、複数選択可能であることを明示し、次だけを選択肢として提示する。

  `おすすめ`、`まとめ系`、`NN`、`NS`、`HR`、`PZR`、`F`、`ギャル`、`綺麗系`、`大人系`、`巨乳`、`スレンダー`、`2回戦以上`、`有名人`、`神コスパ`

- サムネイル画像を受け取っていない場合は、登録作業を開始せず、画像のアップロードを依頼する。
- 公開日はユーザー指定がなければ、登録時点の日本時間の日付を `publishedAt` に使用する。最終確認には必ずこの値を含める。

## サムネイルの検証

- 960×540px（16:9）であることを確認する。
- `public/images/articles/` に保存する。
- ファイル名は英小文字・数字・ハイフンのみで構成し、既存ファイルと重複しない名前にする。
- WebPを優先する。
- 条件を満たさない画像の場合は、ユーザーに差し替えまたは明確な指示を求める。既存画像を上書きしない。

## 登録前の最終確認

必須情報とサムネイルが揃った後、コードを変更する前に必ず次の項目を一覧で提示する。

```text
記事タイトル：
概要：
エリア：
タグ：
通常価格：
販売価格：
Codoc URL：
サムネイル：
記事ID：
公開日：
```

一覧の後に必ず `この内容で登録してよろしいですか？` と質問する。

- ユーザーから「はい」「登録して」「OK」などの明確な承認を受けるまで、ファイル編集・画像配置・Git操作を行わない。
- 承認前の修正希望には、修正された項目から確認を続け、最終確認をやり直す。

## 承認後の登録処理

承認後にのみ、以下を行う。

1. サムネイルを `public/images/articles/` に配置する。
2. `src/content/articles.ts` に一意の記事IDと `publishedAt` を含む記事データを追加する。既存の型と構造に合わせ、必要な型拡張は最小限に行う。
3. タグは `area` に混在させず、指定された本番タグだけを `tags` に登録する。
4. エリア絞り込み、タグ絞り込み、記事カードからのCodoc URLへの遷移に反映されることを確認する。
5. 通常価格がない場合は価格欄を非表示にする。販売価格がない場合も既存仕様と整合する表示にする。
6. 既存記事・既存画像・Hero・デザイン・Webpack/LAN設定は変更しない。
7. `npm run lint` と `npm run build` を実行する。エラー時はGitHubへpushせず、原因を説明して修正する。
8. 成功後、追加記事の表示内容を確認してユーザーへ報告する。

## Git操作

- 記事登録と検証が成功しても、ユーザーの確認を得るまでcommit・pushしない。
- ユーザーが承認した場合のみ、コミットメッセージを `content: add <記事ID>` としてcommit・pushする。
